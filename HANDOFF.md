save/book them, and organizers can publish their own events with a photo and
a map pin. Full organizer toolkit (edit/cancel/duplicate/attendees/marketing-
copy generator), Google Sign-In, Google Places address search, an installable
PWA, and three feature-spec items (Instagram import, cross-platform marketing
copy, Bring Your Own Tickets).

## 3. Two Google integration issues found during recovery — one fixed, one needs your action

### Fixed: a real race-condition bug in `src/google-maps.ts`
The Google Maps script loads with `loading=async`, which means the script
tag's `onload` event fires *before* the `places` sub-library has actually
finished loading — the old code resolved its promise on `onload`, so
`google.maps.places` was sometimes still `undefined` when the map picker
tried to use it, silently falling back to the free OSM search. This bug
predates the drive failure — it was just never caught before because there
was never a working API key to test the success path against until now.
**Fixed** by switching to Google's own ready-callback (`&callback=...` query
param), which only fires once the library is genuinely loaded. Verified:
the map picker now correctly shows "Search any address…" with Google's
autocomplete active instead of the free-search fallback.

### Needs your action: Google Cloud Console configuration
Two real, separate issues surfaced when actually testing with your live
credentials (`AIzaSy...` Maps key, `39719...apps.googleusercontent.com`
Client ID) — these are account/console settings, not code bugs:

1. **Maps**: typing into the location search shows Google's own error
   dialog — *"This page can't load Google Maps correctly. Do you own this
   website?"* This is Google's standard message when either **billing isn't
   enabled** on the Cloud project, or the **Maps JavaScript API** (not just
   Places API) isn't enabled. Fix: console.cloud.google.com → your project →
   Billing (make sure it's linked/active) → APIs & Services → Library →
   confirm both "Maps JavaScript API" and "Places API" show "Enabled".

2. **Sign-In**: the console shows `[GSI_LOGGER]: The given origin is not
   allowed for the given client ID` — a 403. This means the OAuth Client ID's
   **Authorized JavaScript origins** doesn't include the origin you're
   testing from. Fix: console.cloud.google.com/apis/credentials → click the
   OAuth Client ID → Authorized JavaScript origins → add `http://localhost:5173`
   for local dev, and your real deployed frontend URL once you have one. This
   was actually mentioned as a requirement when the key was first requested,
   but apparently wasn't added (or a different origin was used).

Once both of those are fixed on Google's side, no further code changes
should be needed — the button/search UI is already correctly wired to react
to real vs. missing credentials.

## 9. Regenerating the iOS project

The `ios/` Capacitor Xcode project was never hand-written — it's fully
regenerable:
```bash
npm run build
npx cap add ios
npx capacitor-assets generate --ios
npx cap sync ios
```
Then re-apply, if needed: `NSPhotoLibraryUsageDescription` /
`NSCameraUsageDescription` / `ITSAppUsesNonExemptEncryption` in
`ios/App/App/Info.plist`, and the `aps-environment` entitlement for push
(create `ios/App/App/App.entitlements` with `aps-environment: development`
and wire `CODE_SIGN_ENTITLEMENTS = App/App.entitlements` into both Debug and
Release build configs in `project.pbxproj`). Full detail on all of this was
in the previous `BUILD_IOS.md` on the dead drive — ask the user if they want
that walkthrough rewritten, or regenerate it fresh once the iOS project exists
again.
